## 1) Tebligat Durum Enum’ları (hem backend hem frontend aynı konuşsun)

### Tebligat aşaması (en pratik ve yeterli set)

```ts
export type ServiceStatus =
  | "NOT_STARTED"      // hiç hazırlanmadı/çıkmadı
  | "READY"            // tebliğ evrağı hazır (taslak/print)
  | "SENT"             // PTT/elektronik gönderildi
  | "DELIVERED"        // tebliğ edildi (tebliğ tarihi var)
  | "RETURNED"         // iade (sebep var)
  | "MUHTAR"           // muhtara teslim/tebligat kanunu prosedürü
  | "ANNOUNCEMENT"     // ilan yoluyla
  | "FAILED"           // teknik/işlemsel hata
  | "UNKNOWN";         // eski kayıt/elde yok (migrasyon)
```

### İade sebebi (RETURNED için)

```ts
export type ReturnReason =
  | "ADDRESS_NOT_FOUND"
  | "MOVED"
  | "REFUSED"
  | "DECEASED"
  | "COMPANY_CLOSED"
  | "UNCLAIMED"
  | "OTHER";
```

### Kanal (gelişirse iyi olur)

```ts
export type ServiceChannel = "PHYSICAL" | "UETS" | "KEP" | "UNKNOWN";
```

> Minimum viable: `ServiceStatus + deliveredAt + trackingNo` bile iş görür.

## 2) “Kırmızı 1” Uyarı Kural Seti (açık ve deterministik)

### 2.1. Uyarı seviyeleri

```ts
export type AlertLevel = "NONE" | "INFO" | "WARN" | "DANGER";
```

### 2.2. Borçlu kartında uyarı üreten kurallar

Aşağıdakilerden **her biri** bir “issue” üretir. UI’da sağ üstte sayı = issue sayısı.

#### Kimlik / adres eksikleri (kritik)

* `address` yoksa → **DANGER**
* `personType=REAL` ve `tckn` yoksa → **WARN**
* `personType=LEGAL` ve `vkn` yoksa → **WARN**
* `phone/email` ikisi de yoksa → **INFO** (kritik değil ama faydalı)

#### Tebligat kural seti (en kritik)

* status `NOT_STARTED` → **DANGER** (takip var ama tebligat yok)
* status `SENT` ve `sentAt` var ama 7 gün+ geçti & delivered/returned yok → **WARN**
* status `RETURNED` ve `returnReason` yok → **WARN**
* status `DELIVERED` ama `deliveredAt` yok → **DANGER** (tutarsız veri)
* status `FAILED` → **DANGER**

#### Risk sinyalleri (avukat odaklı)

* `riskFlags` içinde `CONCORDAT`/`BANKRUPTCY` → **DANGER**
* `riskFlags` içinde `ADDRESS_SUSPECT` → **WARN**
* `staleDays >= 30` (borçluya dair işlem yok) → **WARN**

#### Malvarlığı araştırması (opsiyonel ama işe yarar)

* “hiç sorgu yapılmadı” & dosya yaşı 15 gün+ → **INFO/WARN** (senin iş akışına göre)

## 3) DebtorRow Data Shape (frontend sözleşmesi)

### 3.1. DTO (listede göstermek için ideal)

```ts
export type DebtorRowDTO = {
  id: string;

  displayName: string;         // "Karadeniz Gıda Ltd. Şti."
  personType: "REAL" | "LEGAL";
  role: "ASIL" | "KEFIL" | "AVAL" | "MIRASCI" | "TEMSilci" | "DIGER";

  identityMasked?: string;     // TCKN/VKN masked
  phoneMasked?: string;
  emailMasked?: string;

  addressShort?: string;       // "Şişli/İstanbul" gibi (tam adres accordion)

  service: {
    status: ServiceStatus;
    channel?: ServiceChannel;
    trackingNo?: string;       // PTT barkod / UETS ref
    sentAt?: string;           // ISO
    deliveredAt?: string;      // ISO
    returnedAt?: string;       // ISO
    returnReason?: ReturnReason;
  };

  assets: {
    vehicle: "YES" | "NO" | "UNKNOWN";
    realEstate: "YES" | "NO" | "UNKNOWN";
    bank: "YES" | "NO" | "UNKNOWN";
    sgkWage: "YES" | "NO" | "UNKNOWN";
    lastQueryAt?: string;      // ISO
  };

  riskFlags: Array<
    | "ADDRESS_SUSPECT"
    | "CONCORDAT"
    | "BANKRUPTCY"
    | "DECEASED"
    | "COMPANY_CLOSED"
  >;

  staleDays?: number;          // borçlu üzerinde işlem yoksa
  quickNote?: string;

  issues: DebtorIssue[];       // “kırmızı 1” buradan türetilir
  alertCount: number;          // issues.length
  alertLevel: AlertLevel;      // max severity
};
```

> Liste performansı için `issues` + `alertCount` backend’den gelsin.
> (İstersen frontend hesaplar ama tutarlılık için backend daha iyi.)

# 2) PUT /debtors/:id/service

**Amaç:** Tebligat durumunu güncellemek (durum, tarih, barkod, iade sebebi vs.)

### Request body

```json
{
  "status": "SENT",
  "channel": "PHYSICAL",
  "trackingNo": "RR123456789TR",
  "sentAt": "2026-01-03T18:10:00+03:00",
  "deliveredAt": null,
  "returnedAt": null,
  "returnReason": null,
  "note": "PTT’ye verildi."
}
```

### Validasyon kuralları (sert)

* `status=DELIVERED` ise `deliveredAt` zorunlu
* `status=RETURNED` ise `returnedAt` + `returnReason` zorunlu
* `status=SENT` ise `sentAt` önerilir, `trackingNo` önerilir
* Tarihler ISO ve gelecekte olamaz (opsiyonel kural)

### Response 200

```json
{
  "id": "deb_123",
  "service": {
    "status": "SENT",
    "channel": "PHYSICAL",
    "trackingNo": "RR123456789TR",
    "sentAt": "2026-01-03T18:10:00+03:00",
    "deliveredAt": null,
    "returnedAt": null,
    "returnReason": null
  },
  "issues": [
    { "code": "MISSING_ADDRESS", "level": "DANGER", "label": "Adres eksik." }
  ],
  "alertCount": 1,
  "alertLevel": "DANGER"
}
```

### Hata

* 400: validasyon hatası (örn deliveredAt yok)
* 404: debtor yok
* 409: statü geçişi çakışması (opsiyonel; aşağıda anlattım)

### Concurrency (öneri)

* Header: `If-Match: <etag>`
* Backend ETag kontrol eder, değişmişse:

  * 412 Precondition Failed

Bu sayede iki kişi aynı anda güncellerse biri diğerini ezmez.

# 3) POST /debtors/:id/queries/run

**Amaç:** UYAP/entegrasyon sorgularını başlatmak (araç/tapu/banka/sgk vs). Asenkron çalışmalı.

### Request body

```json
{
  "types": ["VEHICLE", "REALESTATE", "BANK", "SGK_WAGE"],
  "priority": "NORMAL",
  "reason": "Haciz stratejisi için güncel veri.",
  "idempotencyKey": "case_2026-2_deb_123_20260103_001"
}
```

> `idempotencyKey` şart olsun: kullanıcı iki kez tıklayınca iki job açılmasın.

### Response 202 (Accepted)

```json
{
  "jobId": "job_987",
  "status": "QUEUED",
  "estimated": null
}
```

### Job sonucu nasıl gelir?

İki seçenek:

**A) Polling endpoint**

* `GET /queries/jobs/:jobId`

**B) Push**

* WebSocket / SSE

MVP için polling yeter.

### Örnek job status response 200

```json
{
  "jobId": "job_987",
  "status": "DONE",
  "results": {
    "vehicle": "YES",
    "realEstate": "NO",
    "bank": "UNKNOWN",
    "sgkWage": "YES"
  },
  "updatedAt": "2026-01-03T18:15:00+03:00"
}
```

### Hata

* 400: types boş
* 409: aynı idempotencyKey ile farklı payload
* 429: rate limit (aynı borçluya saatte X sorgu)

## 5) Bonus: Statü geçiş kuralları (409’ı anlamlı kılar)

İstersen statüleri “state machine” gibi kısıtlarız:

* `NOT_STARTED -> READY -> SENT -> DELIVERED`
* `SENT -> RETURNED`
* `RETURNED -> READY` (yeniden tebligat)
* Her yerden `FAILED` olabilir

Geçersiz geçişte:

* `409 Conflict` + “allowedTransitions”

```json
{
  "error": "INVALID_STATUS_TRANSITION",
  "from": "DELIVERED",
  "to": "SENT",
  "allowed": []
}
```

2026-01-03 22:29 (TR)

Evet—**tek ibare + tarih + “yenilenen tebligat” alt akışı** fikri çok daha temiz. Şu anki “yan yana sayaçlar” yerine borçlu satırında tek bir **Tebligat Durumu** ibaresi akıp gitsin; detay ve geçmiş ise açılır panelde yaşasın. Avukat zihni bunu daha hızlı okur.

Aşağıdaki kurgu hem UI’ı sadeleştirir hem de tebligatın gerçek hayat kaosunu (bila, iade, tekrar tebliğ, yeni adres) düzgün taşır.

---

## 1) Borçlu satırında TEK İBARE (ve tebliğ tarihi)

Borçlu kartının sağında **tek bir “Durum Chip’i”** olsun:

### Örnek chip’ler (tek alan, değişerek gider)

* ⚠️ **Tebligat Başlatılmadı**
* ⏳ **Gönderildi** *(PTT: RR…)*
* ✅ **Tebliğ Edildi – 12.01.2026**
* 🔁 **Tebligat Tekrarlandı** *(Yeni adrese)*
* ↩️ **İade – Adres Bulunamadı**
* 🧾 **Bila / Muhtara – 18.01.2026** *(hangi prosedürse)*

**Kural:** Durum chip’i *her zaman tek* ve *en güncel aksiyon gereken durumu* göstermeli.

> Avukat için “şu an ne durumdayız?” sorusunun tek cevabı.

---

## 2) Tebliğ tarihini nasıl göstereceğiz?

Sadece `DELIVERED / MUHTAR / ANNOUNCEMENT` gibi “sonuçlanan” statülerde tarih bas:

* ✅ Tebliğ Edildi — **12 Oca 2026**
* 🧾 Muhtara — **18 Oca 2026**
* 📣 İlan — **03 Şub 2026**

**SENT** durumunda tarih opsiyonel:

* ⏳ Gönderildi — 05 Oca 2026 (isteğe bağlı)

---

## 3) “Bila” geldiğinde açılır pencere + ALTTA YENİ SÜREÇ

Bu kısım tam “gerçek hayata uyum”.

Mantık şu:

* Her borçlu için tebligat **tek bir kayıt** değildir.
* Tebligat **deneme/iterasyon** olarak gider.

### Model: “Tebligat Denemeleri” (attempts)

Borçluya bağlı `serviceAttempts[]` tut:

* Attempt #1: Eski adres → Gönderildi → **Bila/İade**
* Attempt #2: Yeni adres → Hazırlandı → Gönderildi → Bekleniyor
* Attempt #3: ….

### UI davranışı

Borçlu satırında chip:

* 🔁 **Tebligat Tekrarlandı**
  veya eğer son attempt devam ediyorsa:
* ⏳ **Gönderildi** (Attempt #2)

Borçlu satırına tıklayınca açılır panel:

* Üstte “Tebligat Geçmişi” (Attempt’ler timeline)
* Altta “Aktif Tebligat Süreci” (sadece son attempt’in live durumu)

---

## 4) Açılır panel (borçlu alt drawer / accordion) wireframe

Borçlu satırına tıklayınca:

### A) Timeline (Geçmiş)

```
TEBLİGAT GEÇMİŞİ
1) 02.01.2026 — Gönderildi (RR123...)
   10.01.2026 — Bila / Muhtara (sebep: muhatap yok)
   Adres: Bakırköy/İstanbul

2) 12.01.2026 — Tekrar Tebligat (Yeni adres)
   Durum: Hazırlandı
   Adres: Ataşehir/İstanbul
```

### B) Aktif süreç (son attempt)

```
AKTİF TEBLİGAT (Attempt #2)
Durum: Hazırlandı
[ PTT’ye verildi ]  [ UETS gönder ]  [ İade gir ]  [ Tebliğ edildi gir ]
Takip No: —
```

> Böylece “bila” gelince sistem “yeni tebligat aç” diye doğal bir akış sunar.

---

## 5) Sayaçlar ne olacak? (üst şeritte yan yana değil)

Sayaçlar tamamen kalkmasın; ama “yan yana 0/1/0” kafa karıştırıyor.

**Tek özet** yap:

* “Tebligat: 1 borçlu işlem bekliyor”
  ve tıklayınca filtrele.

Ya da en iyi seçenek:

* Üst şeritte sadece:

  * **Borçlu: 5**
  * **Riskli: 5**
  * (opsiyonel) **İşlem Bekleyen: 5**

“0 tebliğ 5 bekleyen 0 iade” gibi üçlü tabloyu kaldır.

## 6) Data modeli (backend DTO değişikliği)

`debtor.service` tek nesne yerine:

```ts
type ServiceAttempt = {
  id: string;
  createdAt: string;
  addressId?: string;
  addressText?: string;

  status: ServiceStatus;
  channel?: ServiceChannel;
  trackingNo?: string;

  sentAt?: string;
  deliveredAt?: string;
  returnedAt?: string;
  returnReason?: ReturnReason;

  note?: string;
};

type DebtorRowDTO = {
  ...
  serviceCurrent: ServiceAttempt;     // en güncel attempt
  serviceHistory: ServiceAttempt[];   // son 3-5 attempt (listede değil, expand’da)
  serviceLabel: string;              // "Tebliğ Edildi — 12.01.2026" (listede hızlı)
};
```

Listede performans için:

* `serviceCurrent` + `serviceLabel` yeter
* `serviceHistory` ancak expand olunca fetch edilebilir:

  * `GET /debtors/:id/service-history`
